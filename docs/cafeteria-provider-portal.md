# Cafeteria Provider Portal

## Routes

The provider portal is separated under `/cafeteria/portal` and uses the `cafeteria.portal.*` route namespace.

- `GET /cafeteria/portal/login` - `cafeteria.portal.login`
- `POST /cafeteria/portal/login` - `cafeteria.portal.login.store`
- `POST /cafeteria/portal/logout` - `cafeteria.portal.logout`
- `GET /cafeteria/portal/dashboard` - `cafeteria.portal.dashboard`
- `GET|POST /cafeteria/portal/scan` - `cafeteria.portal.scan`, `cafeteria.portal.scan.store`
- `GET /cafeteria/portal/transactions` - `cafeteria.portal.transactions.index`
- `GET /cafeteria/portal/transactions/export` - `cafeteria.portal.transactions.export`
- `GET /cafeteria/portal/transactions/export/csv` - `cafeteria.portal.transactions.export.csv`
- `GET /cafeteria/portal/transactions/export/xlsx` - `cafeteria.portal.transactions.export.xlsx`
- `GET /cafeteria/portal/transactions/export/payment-claim` - `cafeteria.portal.transactions.export.payment-claim`
- `GET /cafeteria/portal/ledger` - `cafeteria.portal.ledger.index`
- `GET|POST|PUT|PATCH|DELETE /cafeteria/portal/menus...` - `cafeteria.portal.menus.*`
- `GET|POST /cafeteria/portal/orders...` - `cafeteria.portal.orders.*`
- `GET /cafeteria/portal/reports` - `cafeteria.portal.reports.index`
- `GET|PATCH /cafeteria/portal/profile` - `cafeteria.portal.profile.show`, `cafeteria.portal.profile.update`

## Access Model

Provider portal users use the existing `users` table and must have:

- `status = active`
- `provider_portal_enabled = true` or `user_type = provider`
- an active row in `cafeteria_provider_users`

The portal resolves the selected cafeteria provider from `provider_id`, then session, then the first active assignment. All provider portal queries are scoped to that selected provider.

Route middleware:

- `cafeteria.portal` validates active provider portal access.
- `cafeteria.provider.assigned` requires an active assigned provider.
- `cafeteria.portal.context` shares Inertia `cafeteriaPortal` props only on portal routes.

Provider-only users are redirected away from the admin dashboard to `cafeteria.portal.dashboard`.

## Layout And Navigation

The portal uses `resources/js/Layouts/CafeteriaProviderPortalLayout.tsx`, not the admin/staff layout. The sidebar is limited to dashboard, scan, transactions, ledger, daily menus, food orders, reports, and profile.

## QR Scanning

The provider scan page reuses `ProcessCafeteriaQrScanAction` and `CafeteriaQrScanService`. It accepts the existing stable ID-card QR payload and does not regenerate QR codes or store employee data in the QR token.

Provider scans do not accept `meal_amount` or `requested_subsidy_amount`; the existing cafeteria subsidy rules calculate the payable amounts.

## Ledger

Accepted provider portal scans create a provider ledger entry in `cafeteria_provider_ledger_entries`. The employee subsidy ledger remains handled by the existing cafeteria scan service.

## Menu And Order Scope

Menus, menu items, food orders, order status changes, transactions, reports, and ledger entries are always filtered by the selected provider. Cross-provider route model access returns `404`.

## Transaction Export And Payment Claims

The transactions page supports provider-scoped CSV, XLSX, and PDF exports for billing/payment claims. Export routes use the selected provider context, `cafeteria.portal`, provider assignment middleware, throttling, and `CafeteriaTransactionPolicy::exportProviderTransactions`.

Required export permissions:

- `cafeteria-portal.exportTransactions`
- `provider-cafeteria-transactions.export`
- `provider-cafeteria-payment-claims.export`

CSV, XLSX, and PDF are supported. XLSX uses Laravel Excel. PDF uses DomPDF.

Exports include localized headers, active transaction filters, a safe provider filename, and audit events. CSV includes a UTF-8 BOM for Excel compatibility. Payment claim exports include a summary header with provider, assigned institution, claim period, generated by, generated at, accepted/rejected counts, subsidy payable, employee payable, adjustments, and net payable before transaction rows.

Export rows intentionally exclude QR references, QR/hash fields, national ID, phone number, private file paths, and raw metadata.
